# ReproRelay launch kit

Use this material only after the GitHub Pages landing page is live. It contains public product copy only; it makes no customer, performance, or privacy claims beyond the documented product behaviour.

## Positioning

One sentence:

> ReproRelay turns a vague web bug report into a reproducible, human-reviewed engineering issue.

The proof points to repeat:

- A browser SDK captures the replay, screenshot, browser context, and a client comment.
- The team reviews the evidence before a GitHub issue or agent handoff.
- The project is MIT licensed, self-hosted, and Git-native.
- Input masking and token redaction are part of the default privacy story.

## Product Hunt

Suggested product name:

> ReproRelay — reproducible bug reports for web apps

Suggested description:

> Open-source bug capture for web apps. Turn a client report into a replay, screenshot, browser context, and a human-reviewed issue your team can reproduce.

Suggested first comment:

> Hi Product Hunt — I built ReproRelay for the moment when a client says “it doesn’t work” and engineering has no route, console context, or way to reproduce it.
>
> ReproRelay is a self-hosted, MIT-licensed browser SDK. It captures the report, replay, screenshot, and safe browser context, then keeps a human review step before an issue or agent handoff.
>
> I would especially value feedback from agencies and small product teams: which piece of evidence do you most often have to chase after a bug report arrives?

Ask people to try the demo and leave a useful comment. Do not ask for upvotes.

## Show HN

Suggested title:

> Show HN: ReproRelay – open-source bug capture that turns client reports into reproducible issues

Suggested post:

> I made ReproRelay because “it doesn’t work” is still one of the most expensive bug reports a small product team or agency receives.
>
> The browser SDK gathers the client’s comment with a replay, screenshot, browser context, console signals, and selected network metadata. The dashboard makes the evidence reviewable before it becomes a GitHub issue or reaches an AI coding workflow.
>
> The important constraint for me was keeping the handoff human-gated and privacy-conscious: input masking and token redaction are built into the capture path, and there is a review step before engineering handoff.
>
> It is MIT licensed and self-hostable. I would love feedback from anyone who supports client-facing web applications: what information is usually missing from your incoming bug reports?

## Launch sequence

1. Confirm the landing page and interactive flow work on desktop and mobile.
2. Add the GitHub Pages URL as the repository homepage.
3. Share the public release with five to ten relevant agency or SaaS contacts. Offer setup help, not a generic pitch.
4. Publish the Show HN post and respond personally to every substantive question.
5. Schedule a Product Hunt launch only when you can be present for the discussion. Use a personal maker account and the prepared first comment.
6. Record what users ask for, where they drop off, and whether they submit a first report. Let those signals decide the next product work.
